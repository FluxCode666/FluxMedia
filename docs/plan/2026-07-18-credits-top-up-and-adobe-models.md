# 2026-07-18：积分充值与 Adobe 模型开放范围

## 目标

本次改动覆盖三个相互独立的用户需求：

1. 不在左下角账号菜单展示订阅类型。
2. 在既有一次性积分包之外，支持按金额充值积分；管理员可配置币种、充值区间与每单位货币兑换的积分比例；接入支付宝当面付。
3. Adobe 后端可由管理员配置允许调度的模型，调度与派发两处一致执行该限制。

## 设计约束

- 新充值操作先注册为 UOL operation，再由 server action 和 webhook 作为传输适配器调用。
- 充值订单必须将金额、币种、兑换比例、最终积分数和积分到期时间固化到订单中；支付回调不得依赖会被管理员后来修改的配置。
- 支付回调必须先验签、再比对应用 ID、卖家 PID、订单号、金额和交易状态；订单用交易号唯一索引预占、状态 CAS 租约领取履约，积分发放仍由 `credits_batch(source_type, source_ref)` 唯一约束兜底。
- 支付宝当面付为人民币（CNY）收款方式。一次性积分包可设置 ISO 币种并由 Creem 的对应预建产品结账；不将不受当前支付渠道支持的充值币种暴露为可结账选项。
- Adobe 模型限制属于后端能力边界，候选筛选和派发前校验均须执行，避免仅靠后台 UI 造成绕过。

## 验证

- 充值比例、金额边界、配置解析及支付金额格式使用 DB-free 单测。
- 订单状态机的交易号预占、创建租约、回调金额/卖家校验和积分批次幂等由服务端实现，并由全量类型检查与回归测试覆盖编译与既有业务兼容性。
- Adobe 限制覆盖保存、候选筛选与不允许模型被拒绝的路径。
- 完成后运行受影响包的测试、Biome lint、Turbo typecheck/lint/test。
